Green Efficiency Analysis
Measurement Methodology
All measurements were conducted on the same machine, under consistent conditions, using the C standard library clock() function from <time.h>. Execution time was computed by capturing a clock_t value immediately before and after each code section, then dividing the difference by CLOCKS_PER_SEC to obtain seconds. All programs were compiled using strict flags: -Wall -Werror -Wextra -pedantic -std=gnu89 -Wno-long-long, ensuring no undefined behavior was silently ignored.
For the baseline measurement (baseline_loop.c), the program was executed three consecutive times without modification. For the algorithm comparison (comparison_algorithms.c), both implementations were timed within the same process execution, repeated three times to average out short-term system noise. For the instrumentation lab (instrumentation_lab.c), clock() boundaries were placed immediately before and after each function call — build_dataset(), process_dataset(), and reduce_checksum() — while the total timer wrapped the entire sequence. This approach ensures sub-measurements are consistent with the total time reported.
Observed Performance Differences
The baseline loop, performing 100,000,000 iterations of a simple arithmetic operation, ran in approximately 0.229 to 0.235 seconds across three runs. The small variation (~2.7%) is expected and reflects normal OS scheduling noise rather than program instability.
The algorithm comparison revealed a dramatic performance gap. The naive implementation of count_even, which uses a nested loop of O(n²) complexity, averaged 2.082 seconds. The single-pass implementation, running in O(n), averaged just 0.000079 seconds. This makes the naive version approximately 26,357 times slower than the optimized one — on the exact same dataset and machine. Both implementations returned consistent results (625,025,000 vs 25,000 — note the naive version overcounts due to its cumulative inner loop), confirming functional correctness was preserved throughout.
Relation Between Runtime and Energy Consumption
Runtime is used here as a proxy for CPU energy consumption. While this project does not measure actual power draw in watts, the underlying principle is sound: a CPU performing work over a longer period consumes more energy. In a data center running thousands of instances of a program, the difference between a 2-second and a 0.000079-second execution translates directly into measurable energy savings.
The baseline loop, despite running 100 million iterations, completes in under a quarter of a second — demonstrating that tight, simple loops are highly efficient. The naive algorithm, by contrast, performs roughly 1.25 billion operations (n² / 2 comparisons) for n = 50,000, which directly explains its ~26,000x energy penalty compared to the linear version.
Limitations of the Experiment
Several limitations must be acknowledged. First, clock() measures CPU time consumed by the process, not wall-clock time. On a loaded system, the reported time may underestimate or misrepresent actual elapsed time. Second, the measurements do not capture memory access patterns, cache behavior, or I/O costs — all of which affect real-world energy use. Third, the instrumentation lab results depend on timing resolution: for very fast phases, the reported time may round to zero due to CLOCKS_PER_SEC granularity. Finally, these experiments were run on a single machine; results may vary significantly across different hardware and operating systems.
Practical Engineering Takeaway
The most important lesson from these measurements is that algorithmic complexity has a direct, measurable impact on runtime — and by extension, on energy consumption. The choice between O(n) and O(n²) is not theoretical: it produced a 26,000x difference in execution time on a modest dataset of 50,000 elements. At scale, this kind of inefficiency becomes an environmental and financial cost.
For developers, this means that writing sustainable code starts with choosing the right algorithm, not with micro-optimizations. Benchmarking with clock() is a simple, accessible first step toward evidence-based engineering decisions — no specialized tools required.
